## 二、为什么你还会觉得「像 pending」？

### 原因 ①：`uni.reLaunch` 是异步的

```ts
uni.reLaunch({
  url: "/packageA/pages/login/login",
});
```

👉 它不是立刻跳转
👉 它是 异步微任务 / 原生桥接

此时：

| 状态            | 是否发生 |
| --------------- | -------- |
| reject 执行     | ✅        |
| Promise settled | ✅        |
| 页面开始跳转    | ⏳        |
| UI 线程繁忙     | ⚠️        |

👉 你看到的“卡住”，不是 pending，而是「跳转 + toast 并发」

---

### 原因 ②：你当时 **没有 `await` 拦截器**

```ts
Object.assign(processedConfig, requestInterceptor(processedConfig));
```

这意味着：

| 项目                   | 状态         |
| ---------------------- | ------------ |
| token 刷新             | ❌ 不一定完成 |
| Authorization          | ❌ 可能是旧的 |
| 后端返回 TOKEN_INVALID | ✅            |
| reject                 | ✅            |
| 但请求源头就不稳       | ⚠️            |

👉 Promise 状态是对的，但“上下文是乱的”

---

### 原因 ③：你当时 **没有统一 reject 结构**

```ts
reject(resData);
```

而 `resData` 是：

```ts
{
  code: "...",
  msg: "...",
  data: ...
}
```

很多业务代码会：

```ts
.catch(err => {
  if (err.code) ...
})
```

但如果 上游没处理好，很容易出现：

> ❌ “我以为 reject 了，但它没进 catch”

## 四、一句话终极解释（你可以收藏）

> ❗ Promise 是否 pending，不只看有没有 `reject()`  
> ❗ 还要看：
>
> - reject 是否在 正确时机
> - 是否有 异步跳转 / toast 干扰
> - 上游是否 await 了前置依赖
