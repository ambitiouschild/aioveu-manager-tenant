## ✅ 你现在这段代码的真实行为

```
try {
  const token = await userStore.getValidToken();

    config.header.Authorization = `Bearer ${token}`;
  }
} catch (error: any) {
  if (
    error.message === "NOT_LOGGED_IN" ||

    // ❌ 不在这里跳转
  }
  return Promise.reject(error);

```

### ✅ 实际执行流程是：

| 步骤              | 结果         |
| ----------------- | ------------ |
| `getValidToken()` | ❌ 抛异常     |
| try 中后续代码    | ❌ 不执行     |
| catch             | ✅ 进入       |
| reject            | ✅ 返回       |
| request 后续      | ✅ 不再发请求 |

👉 **请求根本不会发出去**

## ✅ 为什么“不解开”是对的？

### 原因 1️⃣：职责分离（非常重要）

| 层级                     | 职责           |
| ------------------------ | -------------- |
| 请求拦截器               | ✅ 判断是否登录 |
| 请求拦截器               | ✅ reject       |
| `handleTokenExpiredSync` | ✅ 跳转 + UI    |
| request                  | ❌ 不关心       |

✅ **你现在的代码完全符合这个原则**

------

### 原因 2️⃣：避免重复跳转

如果你在这里解开并跳转：

```
uni.reLaunch({ url: "/pages/login/login" });
```

会导致：

| 场景                | 后果       |
| ------------------- | ---------- |
| 并发请求            | ❌ 多次跳转 |
| 拦截器 + 响应拦截器 | ❌ 双重跳转 |
| Promise reject      | ❌ 不明确   |

👉 **你现在这套是“单点兜底”**
